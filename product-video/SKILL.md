# Product Video

把商品参考图直接变成可用于图片转视频的五张独立关键帧。默认先出图，不先生成身份卡、方案文档、Markdown、JSON、联系表或素材包目录。

## 不可覆盖的默认行为

- 零确认：只要能读取并辨认主要商品，就立即开始；不询问商品名、平台、比例、输出目录或是否确认。
- 必须使用已安装的 `$imagegen` Skill 和 Codex 内置 `image_gen` 工具。普通任务不得改用图片 API、CLI 生图、本地模型、SVG、Canvas、Pillow 拼图或占位图。
- 默认生成五张独立的完整 9:16 图片，顺序固定为 `hook → hero → use → detail → ending`。禁止生成五格联系表、拼图、分屏或一图多镜头。
- 严格串行：一次生图调用只处理当前一个镜头；当前图片已经保存或明确记录硬失败后，才能开始下一镜头。禁止并发、批量 `n`、并行子代理生图或一次 prompt 描述五个画面。
- 所有成功图片与唯一 TXT 永远先保存在本地。飞书不可用、未登录、无权限或网络失败都不能阻塞生图和本地落盘。
- 不创建任何运行时 `.md`、`.json`、身份卡、质检报告、失败清单或嵌套素材包目录。最终只允许五张成功图片和一个 UTF-8 BOM TXT；单镜硬失败时图片数可以少于五张，但仍只有一个 TXT。
- 默认尝试把成功图片和一条纯文本发送给当前本机 `lark-cli` 登录的飞书本人。不得硬编码机器人名、用户 ID、群 ID、租户或路径；不得发给其他人或群。
- 用户提供的模糊图仍直接开始，但只保留看得见的轮廓和色块；不得补造文字、Logo、结构、配件、功能、规格或材质。

在动作前完整读取 [references/prompts.txt](references/prompts.txt)。该文件是内部提示词规则，不是运行时产物。

## 1. 自动确定输入、输出与文件名

按以下优先级识别商品图：

1. 用户正文中的明确图片文件路径。
2. 用户正文中的目录路径；按自然文件名排序读取其中常见图片格式，并自动选择主要商品真值图。
3. 当前消息附件暴露的可访问本地路径。
4. 仅存在于会话中的图片附件。

输出目录固定为：

- 文件输入或有本地路径的附件：主要商品真值图的父目录。
- 目录输入：该目录本身。
- 只有会话附件、没有本地路径：当前工作目录。

不要另建子目录。多图跨目录时，以主要商品真值图所在目录为准。

文件前缀固定为：

- 单文件：源文件 stem。
- 目录：目录名。
- 无本地路径：`商品视频-YYYYMMDD-HHmmss`。

清理 Windows 非法文件名字符。开始前检查整批六个目标名；只要任一目标已存在，就给整批前缀追加新的时间戳，绝不覆盖：

```text
<前缀>-01-hook.png
<前缀>-02-hero.png
<前缀>-03-use.png
<前缀>-04-detail.png
<前缀>-05-ending.png
<前缀>-视频提示词.txt
```

保留生图工具返回的实际无损栅格格式；默认优先 PNG。若实际后缀不是 PNG，五张图仍使用相同语义文件名和真实后缀。

生成一个不含身份信息的 `run_id`，只用随 Skill 附带的日志子命令记录 `run_start`：

```text
python <skill_dir>/scripts/deliver_result.py log --run-id <run_id> --event run_start --output-dir <输出目录> --source <源图>
```

不要在首次生图前运行飞书认证或网络 probe；飞书检测统一放到五张图本地落盘以后，避免 CLI 超时拖住首次出图。日志失败时直接继续。脚本会把安全运行记录默认追加到 `D:\UserData\Desktop\生图日志.txt`；公开环境依次回退到 `$HOME/Desktop/生图日志.txt`、`$CODEX_HOME/logs/生图日志.txt` 或 `$HOME/.codex/logs/生图日志.txt`，只有连主目录都不可用时才回退输出目录。共享运行日志不属于本批成果文件。日志不得包含 access token、refresh token、secret、动态 open_id、聊天 ID 或完整授权响应。

## 2. 查看图片并自动锁定商品

- 本地图片先用 `view_image` 查看，再调用内置生图。
- 把最清楚、最正面的同一商品图片作为 `product_truth`。其余同商品视角只补充可见事实。
- 明显不同商品、氛围图、场景图或竞品图只作为 `style_reference`；只取色调、灯光、环境、构图、景深和镜头语言。
- 在内存中记录：可见轮廓、比例、主色、数量、主要部件、Logo/标签位置、包装布局，以及不确定项。不要先向用户展示身份卡。
- 只有附件无法读取、商品轮廓完全不可辨，或多个明显不同商品无法自动选择时才提问。

## 3. 五张图片必须先生成

第一项实质交付必须是图片。不要在第一次 `image_gen` 调用前输出分镜方案、长分析、提示词、TXT 或飞书状态。

每个镜头执行同一闭环：

1. 用 `scripts/deliver_result.py log` 记录 `shot_start`，只记 run_id、源路径、镜头名与目标路径。
2. 基于 `references/prompts.txt` 组装只描述当前镜头的英文 prompt。
3. 调用一次 Codex 内置 `image_gen`。每次调用都明确引用原始商品真值图；风格图只标为 style-only。
4. 工具返回后，立即从工具结果或 `$CODEX_HOME/generated_images/...` 选择本次新生成的文件，并复制到当前镜头的最终目标路径。不得误取旧会话图片。
5. 用 `view_image` 检查当前成品，再按“商品一致、单图 9:16、无叠字、无虚构、无明显畸变”做视觉 QC。
6. 若只是视觉 QC 失败，指出具体缺陷并仅定向重试一次；比较最多两张真实候选，只把较好的一张保存为最终文件。工具级无结果也只在当前镜头内部重试；每个镜头的内置生图调用总数最多三次，其中最多一次是视觉定向重试。
7. 记录 `shot_saved`、`shot_retry` 或 `shot_failed`。有稳定本地商品真值路径时，当前镜头连同重试闭环后才进入下一镜头；仅会话附件无路径时按下方特殊顺序执行。

日志命令形态：

```text
python <skill_dir>/scripts/deliver_result.py log --output-dir <输出目录> --run-id <run_id> --event <事件> --source <源图> --shot <镜头> --status <状态> --output <输出图> --detail <简短安全原因>
```

多张会话图片的工具引用遵循 `$imagegen`：本地目标全部有路径时使用 `referenced_image_paths`；没有路径时使用能覆盖所有目标附件的最小 `num_last_images_to_include`，两者绝不同时传。仅会话附件无路径时，按五个角色各完成一次基础生图并逐张落盘，最多产生五张会话图片，不做视觉重试；第六张会话生图会让原附件超出最近五张范围，因此禁止。只有工具失败且没有产生任何会话图片、原附件仍位于最近五张范围内时，才可重试该基础调用。每次调用只请求当前一张成品，不使用 `n` 生成不同镜头。

如果内置工具只返回会话图片、却没有工具结果路径且 `$CODEX_HOME/generated_images/...` 也没有本次新文件，立即把当前调用记为“不可落盘的工具失败”。不得声称保存成功，也不得用本地重绘伪造结果。

五镜头语义：

- `01-hook`：第一帧就清楚出现商品，用构图、前景或光线吸引注意，不靠虚构功能。
- `02-hero`：全商品最清晰、比例最稳定的英雄展示。
- `03-use`：保守合理的生活方式放置；不能证明的功能一律不演示。
- `04-detail`：只特写参考图中真实可见的材质、轮廓、接缝、边缘或结构，避开模糊小字。
- `05-ending`：完整、稳定、高级的收尾定格，留自然负空间，不加 CTA。

## 4. 只写一个 TXT

五个镜头结束后，在内存中形成纯文本正文。禁止 Markdown 标题符号、表格、代码围栏或 JSON。正文至少包含：

```text
商品视频制作结果
生成时间：...
本地目录：...
商品真值摘要：...
不确定项：...
规格：9:16，5 个镜头，约 10 秒

镜头 01 Hook
图片：...
场景：...
视频提示词：<一条可直接用于 Google Flow、CapCut 或通用图片转视频工具的英文提示词>

...镜头 02 至 05...

统一负面约束：...
生图状态：成功数/失败数及简短原因
```

正文中不要预写“飞书投递”占位行；投递脚本在发送结束后只追加一行真实最终状态，避免重复。

用 UTF-8 BOM 保存，确保 Windows 记事本可直接打开。单镜失败写回此 TXT，不另建 `failed.txt`。

## 5. 本地成果完成后再投递飞书

调用投递脚本；`--image` 只传实际成功且已落盘的图片，按 01 到 05 顺序重复，允许 0 至 5 张。纯文本正文经 stdin 传入，不上传 Markdown 文件：

```text
python <skill_dir>/scripts/deliver_result.py deliver \
  --output-dir <输出目录> \
  --prefix <整批前缀> \
  --run-id <run_id> \
  --image "01-hook=<01路径>" \
  --image "02-hero=<02路径>" \
  --image "03-use=<03路径>" \
  --image "04-detail=<04路径>" \
  --image "05-ending=<05路径>" \
  --result-text-stdin
```

某镜头硬失败时直接省略对应 `--image`，保留其他镜头的显式角色，不得把后续图片向前重编号。五张都失败时不传 `--image`，仍写 TXT 并尝试发送纯文本。

投递脚本负责：

- 确保图片仍在输出目录，并写出唯一 BOM TXT。
- 动态运行 `lark-cli auth status --json --verify`；不执行 `auth login`，不修改登录状态。
- 若当前用户身份经服务器验证、具备消息权限且能动态取得本人 ID，优先用 `--as user` 发给本人。
- 仅当用户身份不可用或某未发送项失败，且本机 bot 确实 ready 时，才动态用 `--as bot` 补发尚未成功的项目。
- 不读或硬编码“小白”、open_id、chat_id 或任何固定机器人；没有可验证的本人目标时只保存本地。
- 图片消息与纯文本消息分开发送，五张图在前、纯文本在后；传给 lark-cli 的媒体路径必须是当前 cwd 下的相对路径。
- 使用内容哈希幂等键；已成功项不得因身份切换而重发。
- 网络错误最多安全重试一次；权限、认证、参数或收件人错误不重试。
- 把 `user`、`bot`、`mixed` 或 `local_only` 和简短原因补进本地 TXT 与日志，但不记录动态身份 ID。

调用该 Skill 只授权默认投递给当前登录本人。任何其他收件人、群聊、公开文档或云盘位置都必须另行确认，不能从历史配置猜测。

## 6. 生图能力不可用时

如果当前 Codex 会话没有内置 `image_gen`，或工具连续失败且没有真实图片结果：

- 记录 `imagegen_unavailable` 或具体工具错误到安全日志。
- 不调用需要 API Key 的 CLI，不使用本地大模型，不用本地绘图库伪造成品，不创建占位图。
- 仍生成唯一 BOM TXT，写入五条已经准备好的视频/关键帧提示词和失败原因。
- 本机飞书可用时只发送这条纯文本；否则只保存在原目录。
- 清楚告诉用户当前会话没有完成真实生图，并给出 TXT 路径。

## 7. 完成规则

实际发生任何内置生图调用后，遵守 `$imagegen` 的会话展示规则：后续保存、QC、重试、日志和飞书投递都静默用工具完成，不在最后追加下载说明、图片摘要、追问或额外文字。没有发生真实生图时，才用一句简洁说明报告失败原因和 TXT 本地路径。
