# 商品视频导演台

把商品图变成一条有因果、有镜头变化、能用首尾帧连续生成的五段竖屏广告。不要把任务理解成“五张互不相干的商品海报”。

开始前完整读取 [references/prompts.txt](references/prompts.txt)。该文件是内部导演和提示词规则，不是运行时产物。

## 默认交付合同

- 零确认：只要主要商品可辨认就直接开始，不询问商品名、平台、比例、目录或方案确认。
- 必须使用已安装的 `$imagegen` Skill 和 Codex 内置 `image_gen`。不得改用图片 API、CLI 生图、本地模型、SVG、Canvas、Pillow、拼图或占位图。
- 默认生成 6 张完整 9:16 共享锚点帧，组成 5 段视频：`01→02`、`02→03`、`03→04`、`04→05`、`05→06`。
- 严格串行，一次生图只生成当前一张完整图片。禁止并行生图、联系表、拼图、分屏或一次 prompt 描述多个画面。
- 第一张图前必须先在内存中完成整条导演表并创建项目目录；导演表不落盘，也不先发给用户。
- 每次只引用商品真值图、必要的风格参考和上一张合格锚点；不要把全部旧锚点都塞给生图工具。
- 所有本地成果完成后才检测飞书。飞书不可用不能阻塞生图或本地落盘。
- 成果目录只允许 `01.png` 至 `06.png` 和 `导演台.txt`；硬失败时图片可少，但仍只有一个 TXT。禁止运行时 Markdown、JSON、身份卡、报告和失败清单。
- 默认把成功图片和同一份纯文本发给当前本机 `lark-cli` 登录的飞书本人。不得硬编码机器人名、用户 ID、群 ID、租户或收件人。
- 模糊图只保留可见轮廓和色块；不得补造文字、Logo、结构、配件、功能、规格或材质。

## 1. 锁定输入并新建项目目录

按以下优先级找商品图：

1. 用户正文中的文件路径。
2. 用户正文中的目录路径；按自然文件名读取常见图片并自动选主要商品真值图。
3. 当前消息附件的可访问本地路径。
4. 仅存在于会话里的图片附件。

把最清楚、最正面的同一商品图片设为 `product_truth`。同商品其他角度只补充可见事实；场景图、竞品图和情绪图只能做 `style_reference`。

输出根目录：

- 文件输入或本地附件：商品真值图父目录。
- 目录输入：该目录本身。
- 只有会话附件：当前工作目录。

生成不含身份信息的 `run_id`，在第一次生图前调用初始化命令。脚本负责创建 `<商品名>_导演台_YYYYMMDD-HHmmss`；冲突时只追加 `-02`、`-03`，绝不覆盖：

```text
python "<skill_dir>/scripts/deliver_result.py" init --run-id "<run_id>" --source "<源图或目录>" --base-dir "<输出根目录>"
```

仅会话附件没有路径时省略 `--source`，并传 `--product-name 商品`。从命令返回的 JSON 读取 `project_dir`，不要把 JSON 保存为文件。

中央安全日志默认追加到 `D:\UserData\Desktop\生图日志.txt`；公开环境依次回退到桌面、`$CODEX_HOME/logs` 或 `$HOME/.codex/logs`。日志不得写 token、secret、open_id、chat_id 或完整认证响应。

## 2. 先做内部产品圣经与导演表

用 `view_image` 查看所有本地参考图。在内存中锁定：

- 商品外形、比例、数量、主色、可见材质、部件、开口、接缝、Logo/标签位置和包装布局。
- 不确定、模糊或被遮挡的部分；后续始终保持中性。
- 同一视觉世界的色板、场景、时间、主光方向、色温、表面、镜头质感和尺度。
- 一句创意母题，以及从开场状态到最终落版的一条可见因果动作链。

风格参考只决定色调、灯光、环境、构图、景深和镜头语言。不得复制其中的商品、人物、品牌、文字或包装。

先在内存中提出三套机制不同的创意，不得只是同一摆拍换三种风格。按商品忠实、视觉冲击、因果清楚、连续可实现和品类相关各 1–5 分评分；含两次以上“泛光扫＋推近＋英雄落版”的方案直接淘汰，只执行最高分方案。

导演表以 5 个视频段为主体，不以单张图片为主体：

```text
段号｜叙事任务｜首帧静态状态｜尾帧静态状态｜本段新增事实
唯一主动作｜唯一可实现运镜｜可选的被动环境余波｜落点｜与下一段的交接
```

再从五段的首尾状态推导 6 个共享锚点。锚点只记录静态连续性：商品位置、朝向、状态、接触，人物/手位置，场景轴线，镜头位置与取景，主光和前中后景；不要给锚点另配一套运镜。

六个锚点不是六张换角度展示图。它们必须形成：

```text
01 开场状态 → 02 被第一动作改变的结果 → 03 因果继续发展
→ 04 真实证据或视觉回报 → 05 问题/动作得到解决 → 06 稳定品牌记忆点
```

按商品事实调整每一步；无法验证使用方式时，用灯光、前景、相机和环境的物理变化推进，不虚构功能。

## 3. 镜头逻辑与去重硬规则

连续性合同：

- 第 N 段的尾帧就是第 N+1 段的首帧；不要为同一交接重复生成两张近似图。
- 相邻锚点锁定同一商品、场景世界、时间、主光方向和合理空间关系。
- 商品或人物的运动方向不得无故反转；动作不能在下一段重新开始。
- 每段只选一种有动机的交接：动作匹配、形状/颜色匹配、同方向遮挡、光扫、焦点转移或明确硬切。
- 场景跳变只有在前一锚点已经设计出可剪切的遮挡、光闪、运动模糊或完整落点时才允许。

镜头差异合同针对五个视频段，而不是共享锚点：

- 每段首尾状态的变化必须能由一个主动作和一种连续运镜物理实现；不能同时跳景别、机位高度和水平轴线。
- 相邻视频段至少改变以下五项中的两项：叙事任务、运镜、景别变化趋势、机位高度/水平角度、景深关注层。
- 五段至少使用四种不同的 `叙事任务＋运镜＋景别变化趋势＋机位` 组合。
- 俯拍最多一次；背面只有商品背部确有信息时才允许；居中英雄构图最多两次。
- 不得机械生成前、后、左、右、上、下六个角度；每个机位必须服务于当前叙事任务。
- 非相邻锚点不得无动机地复刻同一景别、机位、商品朝向和画面位置；只有最终有意回扣开场时可重复一项构图关系。
- 每张只出现一个商品实例。除商品本身真实存在的屏幕外，禁止镜面、完整倒影、额外屏幕画面、前景复制或背景复制形成第二个可辨识商品。

视觉质量合同：

- 每张有且只有一个清楚的视觉主意、明确焦点和可读商品轮廓。
- 至少利用前景、中景、背景中的两层制造纵深；避免商品反复站在空白台座中央。
- 允许大胆但受控的商业美术：有动机的色彩对比、建筑尺度、材质环境、体积光或环境微粒；不得靠爆炸、漂浮拆件、无依据特效或堆满道具制造“惊艳”。
- 商品必须有真实接触、阴影、尺度和材质响应；不可漂浮、融化、变形或被手遮住关键信息。

## 4. 串行生成 6 张共享锚点

如果 `product_truth` 只有会话附件、没有任何本地路径，必须先解决滚动图片窗口，不能直接开始六锚点：

1. 用一次 `image_gen` 和能覆盖商品附件的最小 `num_last_images_to_include`，制作一张内部 `product_truth_proxy.png`。在 prompt 中按可见内容明确指出哪张是商品真值；同批被动包含的风格图在这一步完全忽略。要求忠实保留可见轮廓、比例、颜色、部件数和标签布局，模糊文字继续模糊；中性背景，不做创意场景，不补全细节。
2. 把真实 PNG 结果保存到系统临时目录，不放进项目目录；用 `view_image` 对照会话中可见的原附件做身份 QC，最多定向重试一次。
3. 后续六个锚点一律把该本地代理作为身份传输参考，并在文字锁中继续以原附件可见事实为最高真值。会话里的风格参考只转写为色板、灯光、环境和镜头语言，不再占图片窗口。
4. 全部投递结束后删除临时代理。代理失败则停止并走“生图不可用”，不得靠上一张创意锚点冒充永久商品真值。

每个锚点执行同一闭环：

1. 用 `scripts/deliver_result.py log` 记录 `anchor_start`。
2. 按 `references/prompts.txt` 只组装当前锚点的英文生图 prompt。
3. 锚点 01 引用有本地路径的 `product_truth`（或临时代理）和可选单张本地风格参考；锚点 02–06 引用同一永久身份参考、上一张合格锚点和可选单张本地风格参考。明确上一帧只控制世界、空间和动作承接，不得复制其构图。
4. 调用一次内置 `image_gen`，立即把本次真实 PNG 结果保存为项目目录中的 `01.png` 至 `06.png`。工具若没有提供可落盘的真实 PNG，就记为当前锚点失败；不得给 JPEG/WebP 直接改成 `.png` 冒充。
5. 用 `view_image` 检查商品真值、单图质量、与上一锚点的承接和镜头差异。
6. QC 失败时指出一个具体缺陷，只定向重试当前锚点一次；保留更好的真实候选。工具无结果可额外安全重试一次。第一套创意的基础生成、工具重试、单图 QC 和后续跨镜修复合计，每个锚点最多三次。
7. 记录 `anchor_saved`、`anchor_retry` 或 `anchor_failed` 后才能进入下一锚点。

当前锚点硬失败时立即停止后续生图；没有这一共享端点，继续生成只会制造断链。保留已经成功的前序锚点，并按“生图不可用”完成唯一 TXT。

日志命令：

```text
python "<skill_dir>/scripts/deliver_result.py" log --output-dir "<项目目录>" --run-id "<run_id>" --event "<事件>" --source "<源图>" --shot "<01至06>" --status "<状态>" --output "<输出图>" --detail "<安全短原因>"
```

六锚点阶段只使用 `referenced_image_paths`，因为永久商品真值此时必须已有本地路径。只有创建临时身份代理的那一次允许使用 `num_last_images_to_include`；两种引用机制绝不在同一次调用中混用。

## 5. 完成整条跨镜 QC

六张图全部生成后，再整体检查一次：

- 01 到 06 是否真的推进了一个事件，而不是重复摆拍。
- 相邻帧是否能解释中间会发生的一个短动作。
- 商品、左右方向、人物/手、主光和场景是否连续。
- 是否出现近似机位、上下前后机械轮换、产品复制、完整倒影或构图复刻。
- 最终 06 是否是前面动作的结果，而不是突然换成另一张海报。

发现单个断接或重复时，只重做最早出现问题的后一个锚点；它必须同时满足前一段尾态和后一段首态。随后依次复核后续锚点：只有下一个锚点不再适配时才向前传播重做，不能留下依赖旧状态的断链。跨镜修复只允许一轮，且仍计入第一套创意每锚点最多三次的上限。

如果“因果推进”或“视觉冲击”低于 4 分并同时影响两段以上，判定为创意母题失败，不要用局部光效修补。最多回退一次到第二名创意，在系统临时目录从 01 到 06 串行生成候选；第二套每个锚点只调用一次、不做单图重试，因此整次运行每个锚点最多四次。只有第二套完整且整条评分更高时才安全替换项目中的六张图，否则删除候选并保留第一套；第二套仍不合格时停止。

## 6. 只写一个专业 TXT

默认用简洁中文写提示词；用户明确要求其他语言时才切换。禁止 Markdown、表格、JSON、商品分析、真值摘要、不确定项、绝对路径、统一负面词、成功统计或飞书状态。

TXT 必须严格使用以下结构，每个编号只对应一条可直接复制的视频提示词：

```text
项目：<商品名>
规格：9:16｜5 段｜每段 4–6 秒

01｜<短镜头名>
首尾帧：01.png → 02.png
提示词：<一条短段落>

02｜<短镜头名>
首尾帧：02.png → 03.png
提示词：<一条短段落>

03｜<短镜头名>
首尾帧：03.png → 04.png
提示词：<一条短段落>

04｜<短镜头名>
首尾帧：04.png → 05.png
提示词：<一条短段落>

05｜<短镜头名>
首尾帧：05.png → 06.png
提示词：<一条短段落>
```

每条提示词只描述两帧之间的运动，不重复解释图片已经给出的外观。固定顺序：`主体/环境的一个可见动作 → 一种运镜 → 承接惯性与落点 → 商品防漂移约束`。不得在一段内同时推拉摇移，不写硬编码的 0–3 秒时间轴，不塞第二个事件。

格式标签始终保留中文，只有 `提示词：` 后的正文按用户要求切换语言。除可选的单行 `生图：失败（简短原因）` 外，脚本会拒绝任何额外分析或状态行。此时只在内存中组装正文，不提前创建 `导演台.txt`；投递脚本是唯一写入者，并用 UTF-8 BOM 一次落盘，确保 Windows 记事本可直接打开。

## 7. 本地完成后投递飞书本人

调用投递脚本；`--image` 只传真实成功图片，保留原编号，不向前补位。下面是 PowerShell 形态；把内存中的完整正文放进 here-string，经 stdin 交给脚本，由脚本首次创建 TXT：

```text
@'
<内存中的 TXT 正文>
'@ | python "<skill_dir>/scripts/deliver_result.py" deliver --output-dir "<项目目录>" --run-id "<run_id>" --image "01=<01路径>" --image "02=<02路径>" --image "03=<03路径>" --image "04=<04路径>" --image "05=<05路径>" --image "06=<06路径>" --result-text-stdin
```

脚本负责：

- 确保六张图片和唯一 BOM TXT 位于项目目录，不覆盖既有文件。
- 动态运行 `lark-cli auth status --json --verify`；不登录、不修改认证状态。
- 优先以已验证的当前用户身份发送给本人；只有用户身份不可用或未发项目失败且本机 bot 确实 ready 时，才动态补发。
- 图片按 01–06 发送，纯文本最后发送；飞书状态只进 JSON stdout 和中央日志，不污染 `导演台.txt`。
- 使用内容哈希幂等键；网络错误最多重试一次，权限、认证和收件人错误不重试。

调用本 Skill 只授权投递给当前登录本人。其他人、群聊、公开文档或云盘位置必须另行确认。

## 8. 生图不可用

当前会话没有内置 `image_gen`、会话附件无法建立可靠的本地身份代理，或工具连续失败且没有真实结果时：

- 不调用需要 API Key 的 CLI，不使用本地模型或绘图库伪造成品，不创建占位图。
- 仍按 01–05 结构准备五条备用提示词，并在规格行后只加一行准确状态，例如 `生图：失败（04.png–06.png 未生成；提示词仅作备用）`；不得把缺失首尾帧说成可直接使用。
- 正文仍由 `deliver` 一次写成唯一 `导演台.txt`；已有成功锚点按原编号传入，缺失项不传。
- 飞书可用时只发送纯文本，否则保留本地。
- 没有发生真实生图时，最后用一句话报告原因和 TXT 路径。

实际调用过内置生图后，遵守 `$imagegen` 的会话展示规则：后续保存、QC、重试、日志和飞书投递静默完成，最终不追加下载说明、图片摘要或追问。
